# 컴포넌트 설계 패턴

React 앱이 커질수록 **컴포넌트를 어떻게 나누고 데이터를 흘려보낼지**가 중요해진다.  
자주 쓰이는 설계 패턴과 원칙을 정리한다.

---

## 1. 컨테이너 vs 프레젠테이션

| | 컨테이너 (Smart) | 프레젠테이션 (Dumb) |
| --- | --- | --- |
| **역할** | 데이터 fetch, State 관리 | UI만 렌더 |
| **Hooks** | `useState`, `useEffect` 등 사용 | 거의 없음 |
| **Props** | 하위에 데이터·핸들러 전달 | 받은 Props로 표시 |

```jsx
// 컨테이너
function UserPageContainer() {
  const [user, setUser] = useState(null);

  useEffect(() => {
    fetchUser().then(setUser);
  }, []);

  if (!user) return <Loading />;
  return <UserProfile name={user.name} email={user.email} />;
}

// 프레젠테이션
function UserProfile({ name, email }) {
  return (
    <div>
      <h1>{name}</h1>
      <p>{email}</p>
    </div>
  );
}
```

UI 재사용·테스트가 쉬워진다.

## 2. 합성 (Composition)

상속보다 **컴포넌트를 조합**하는 방식을 권장한다.

```jsx
function Dialog({ title, children, footer }) {
  return (
    <div className="dialog">
      <header>{title}</header>
      <main>{children}</main>
      <footer>{footer}</footer>
    </div>
  );
}

<Dialog
  title="삭제 확인"
  footer={<button>확인</button>}
>
  정말 삭제하시겠습니까?
</Dialog>
```

### Slot 패턴

특정 영역을 Props로 받아 유연하게 채운다.

```jsx
function Layout({ header, sidebar, content }) {
  return (
    <div className="layout">
      <aside>{sidebar}</aside>
      <div>
        {header}
        {content}
      </div>
    </div>
  );
}
```

## 3. Props Drilling과 해결

Props가 여러 단계를 거쳐 전달되는 것을 **Props drilling**이라 한다.

```text
App (user)
 └── Layout (user)
      └── Sidebar (user)
           └── Avatar (user)  ← 3단계 전달
```

### 해결 방법

| 방법 | 적합한 경우 |
| --- | --- |
| **컴포넌트 합성** | 중간 컴포넌트가 user를 쓰지 않으면 children으로 우회 |
| **Context** | 테마, 인증 등 앱 전역 값 |
| **상태 관리 라이브러리** | Zustand, Redux 등 |

```jsx
// 합성으로 drilling 줄이기
function App() {
  const user = useUser();
  return (
    <Layout sidebar={<Sidebar user={user} />} />
  );
}
```

## 4. State 배치 원칙

> **State는 그것을 필요로 하는 컴포넌트들의 가장 가까운 공통 부모에 둔다.**

```text
❌ App에 모든 State
App (모든 State)
├── Header
├── Sidebar
└── Main

✅ 필요한 범위에만
App
├── Header
└── Main
    └── SearchPanel (query State)  ← 여기서만 쓰이면 여기에
```

자주 바뀌는 State를 트리 상단에 두면 **불필요한 리렌더**가 늘어난다.

## 5. 단일 책임 원칙

한 컴포넌트가 **한 가지 일**만 하도록 나눈다.

```jsx
// ❌ 한 컴포넌트에 fetch + 폼 + 리스트
function UserManager() { ... }

// ✅ 역할 분리
function useUsers() { ... }        // 데이터
function UserForm({ onSubmit }) { ... }
function UserList({ users }) { ... }
function UserManager() { ... }     // 조합만
```

## 6. React.memo와 선택적 리렌더

Props가 바뀌지 않으면 리렌더를 건너뛴다.

```jsx
const ExpensiveList = React.memo(function ExpensiveList({ items }) {
  return (
    <ul>
      {items.map((item) => (
        <li key={item.id}>{item.name}</li>
      ))}
    </ul>
  );
});
```

주의:
- **모든 컴포넌트에 memo를 쓰지 않는다**
- Props로 **객체·함수**를 매번 새로 만들면 memo가 무의미해진다 → `useMemo`, `useCallback`과 함께 사용

## 7. 관심사 분리: 로직을 Hook으로

UI와 비즈니스 로직을 분리한다.

```jsx
function usePosts() {
  const [posts, setPosts] = useState([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetchPosts()
      .then(setPosts)
      .finally(() => setLoading(false));
  }, []);

  return { posts, loading };
}

function PostPage() {
  const { posts, loading } = usePosts();

  if (loading) return <Spinner />;
  return <PostList posts={posts} />;
}
```

---

## 8. 에러·로딩·빈 상태 처리

데이터 의존 UI는 **3가지 상태**를 명시적으로 다룬다.

```jsx
function PostList({ posts, loading, error }) {
  if (loading) return <Spinner />;
  if (error) return <ErrorMessage error={error} />;
  if (posts.length === 0) return <EmptyState />;

  return (
    <ul>
      {posts.map((post) => (
        <PostItem key={post.id} post={post} />
      ))}
    </ul>
  );
}
```

## 10. 안티 패턴 (피할 것)

| 안티 패턴 | 문제 |
| --- | --- |
| 거대한 God Component | 유지보수 어려움 |
| Props 직접 수정 | 단방향 흐름 깨짐 |
| useEffect 남용 | 불필요한 실행, 버그 |
| 인덱스를 key로 남용 | 리스트 버그 |
| 과도한 Context | 전역 리렌더 |
| 성급한 최적화 | memo/useMemo 남용 |

## 전체 개념 연결도

```text
[개요] React = 컴포넌트 기반 UI 라이브러리
   ↓
[컴포넌트] UI를 재사용 단위로 분리
   ↓
[JSX] 컴포넌트 안에서 UI 기술
   ↓
[Props/State] 데이터 입력과 내부 상태
   ↓
[렌더링] State 변경 → Virtual DOM → DOM 패치
   ↓
[이벤트] 사용자 입력 → setState
   ↓
[Hooks] State·effect·로직 재사용
   ↓
[설계 패턴] 확장 가능한 구조
```
