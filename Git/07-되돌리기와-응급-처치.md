# 되돌리기와 응급 처치

실수는 누구나 한다. 프론트엔드 개발에서 자주 겪는 Git 사고와 복구 방법을 정리한다.

---

## 상황별 빠른 참조

| 상황 | 명령어 |
| --- | --- |
| 파일 수정 취소 (unstaged) | `git restore <파일>` |
| add 취소 | `git restore --staged <파일>` |
| 직전 커밋 메시지 수정 (push 전) | `git commit --amend` |
| 직전 커밋 취소, 변경 유지 | `git reset --soft HEAD~1` |
| 직전 커밋 취소, 변경도 삭제 | `git reset --hard HEAD~1` |
| 이미 push한 커밋 되돌리기 | `git revert <커밋>` |
| 잘못된 브랜치에 커밋함 | `git cherry-pick` / `git stash` |

---

## git reset — 로컬 히스토리 되돌리기

```bash
git reset --soft HEAD~1   # 커밋만 취소, 변경은 staged
git reset --mixed HEAD~1  # 커밋 취소, 변경은 unstaged (기본값)
git reset --hard HEAD~1   # 커밋 + 변경 모두 삭제 ⚠️
```

```text
HEAD~1 = 바로 이전 커밋
HEAD~2 = 2개 전 커밋
```

### ⭐ push한 커밋에 reset --hard 금지

공유 브랜치에서 `reset --hard` 후 `push --force`하면 **팀원 히스토리가 깨진다**.  
이미 push한 경우 `git revert`를 쓴다.

---

## git revert — 안전한 되돌리기

새 커밋으로 **이전 변경을 취소**한다. 히스토리를 지우지 않아 공유 브랜치에 안전.

```bash
git revert abc1234
git revert HEAD        # 직전 커밋 되돌리기
```

```text
main: A --- B --- C --- R
                    ↑     ↑
                  문제   revert 커밋 (B의 변경 취소)
```

프로덕션 hotfix 롤백에 자주 쓴다.

---

## 잘못된 브랜치에 커밋함

### 방법 1: cherry-pick

```bash
# main에 실수로 커밋함
git log --oneline -1   # 커밋 해시 확인 (abc1234)

git checkout feature/correct-branch
git cherry-pick abc1234

git checkout main
git reset --hard HEAD~1   # main에서 제거 (push 전에만!)
```

### 방법 2: stash

```bash
git stash
git checkout feature/correct-branch
git stash pop
git add .
git commit -m "feat: 올바른 브랜치에 커밋"
```

---

## 실수로 main에 push함

### 아직 아무도 pull 안 했을 때 (드묾)

팀에 알리고, 규칙에 따라 revert 또는 reset.

### 이미 팀원이 pull 했을 때

```bash
git revert <잘못된_커밋>
git push
```

**force push는 최후의 수단**이고, 팀 전체 합의가 필요하다.

---

## git reflog — 삭제한 커밋 복구

`reset --hard`로 날린 커밋도 **일정 기간** 복구 가능.

```bash
git reflog
# abc1234 HEAD@{1}: commit: feat: 중요한 작업

git checkout abc1234
# 또는
git reset --hard abc1234
```

---

## force push — 주의해서 사용

```bash
git push --force-with-lease   # ✅ 권장 (원격이 예상과 다르면 거부)
git push --force              # ⚠️ 위험 (무조건 덮어씀)
```

`--force-with-lease`는 **내가 마지막으로 본 원격 상태**와 같을 때만 push한다.  
다른 사람이 먼저 push했으면 실패해서 사고를 막아준다.

**사용 가능한 경우:**

- 혼자 쓰는 feature 브랜치에서 rebase 후 push
- PR squash 전 로컬 정리

**금지:**

- main / develop 등 공유 브랜치

## 민감 정보를 커밋함 (.env 등)

```bash
# 1. 파일을 .gitignore에 추가
echo ".env" >> .gitignore

# 2. Git 추적에서 제거 (파일 자체는 유지)
git rm --cached .env
git commit -m "chore: .env를 Git 추적에서 제거"

# 3. 이미 push했다면 — 히스토리에서 완전 삭제 필요
# git filter-repo 또는 BFG Repo-Cleaner
# + API 키 즉시 재발급!
```

`.env`가 올라갔으면 **키를 무효화**하는 것이 최우선이다. Git에서 지우는 것만으로는 부족하다.

## merge/rebase 망가졌을 때

```bash
git merge --abort
git rebase --abort
```

대부분 작업 전 상태로 돌아간다. 당황하지 말고 `--abort`부터 시도.

## 프론트엔드 응급 시나리오

### "배포 후 버그 — 방금 merge 롤백"

```bash
git checkout main
git pull
git revert <merge-commit-hash> -m 1
git push
# -m 1: merge commit에서 main 쪽 부모 기준 revert
```

### "node_modules를 실수로 add함"

```bash
git restore --staged node_modules
# .gitignore에 node_modules/ 확인
```

### "dist 폴더가 커밋됨"

```bash
git rm -r --cached dist/
echo "dist/" >> .gitignore
git commit -m "chore: dist 빌드 결과물 추적 제거"
```

---

## 정리

| 명령어 | 위험도 | 용도 |
| --- | --- | --- |
| `git restore` | 낮음 | 파일/add 되돌리기 |
| `git reset --soft` | 낮음 | 커밋 취소, 변경 유지 |
| `git reset --hard` | 높음 | 로컬 변경 완전 삭제 |
| `git revert` | 낮음 | 공유 브랜치 안전 롤백 |
| `git reflog` | 낮음 | 잃어버린 커밋 찾기 |
| `push --force` | 매우 높음 | 공유 브랜치 금지 |
