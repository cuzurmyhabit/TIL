# 브랜치 전략

브랜치는 **독립적인 작업 공간**이다. 프론트엔드 팀에서는 기능·버그·배포를 브랜치로 분리하는 것이 기본이다.

---

## 브랜치 기본 명령어

```bash
# 브랜치 목록
git branch

# 원격 포함
git branch -a

# 새 브랜치 생성 + 이동
git checkout -b feature/login-ui
# Git 2.23+
git switch -c feature/login-ui

# 브랜치 이동
git checkout main
git switch main

# 브랜치 삭제
git branch -d feature/login-ui       # merge된 브랜치
git branch -D feature/login-ui       # 강제 삭제

# 원격 브랜치 삭제
git push origin --delete feature/login-ui
```

---

## 브랜치 네이밍 규칙

팀마다 다르지만, 프론트엔드에서 흔한 패턴:

| 접두사 | 용도 | 예시 |
| --- | --- | --- |
| `feature/` | 새 기능 | `feature/dark-mode` |
| `fix/` | 버그 수정 | `fix/header-overflow` |
| `hotfix/` | 긴급 프로덕션 수정 | `hotfix/payment-crash` |
| `chore/` | 설정·의존성 | `chore/upgrade-react-19` |
| `refactor/` | 리팩터링 | `refactor/auth-hooks` |

```text
✅ feature/user-profile-modal
✅ fix/mobile-nav-zindex
❌ jisoo-branch
❌ test123
```

이슈 트래커와 연동하면 추적이 쉽다.

```text
feature/PROJ-123-login-form
fix/PROJ-456-image-loading
```

## main vs develop

| 브랜치 | 역할 |
| --- | --- |
| `main` | **배포 가능한** 안정 코드. PR merge 대상 |
| `develop` | (선택) 다음 릴리스 통합 브랜치 |
| `feature/*` | 개별 기능 개발 |

### Trunk-based (요즘 많이 씀)

```text
main ←── feature/A (짧은 수명, 자주 merge)
     ←── feature/B
     ←── fix/C
```

- feature 브랜치를 **1~2일 안에** merge
- main이 항상 deployable
- 스타트업·애자일 팀에서 흔함

### Git Flow (전통적)

```text
main ───────────────────────────── (릴리스)
  \
develop ─── feature/A ─── feature/B ───
```

- 릴리스 주기가 길고 QA 단계가 명확할 때
- 프론트엔드 단독 팀보다 **대규모 제품**에서 더 흔함

## 브랜치 작업 흐름 (실무 표준)

```bash
# 1. main 최신화
git checkout main
git pull

# 2. feature 브랜치 생성
git switch -c feature/search-bar

# 3. 작업 + 커밋 (여러 번)
git add .
git commit -m "feat: SearchBar UI 구현"
git commit -m "feat: 검색 API 연동"

# 4. push
git push -u origin feature/search-bar

# 5. GitHub에서 PR 생성 → 리뷰 → merge

# 6. 로컬 정리
git checkout main
git pull
git branch -d feature/search-bar
```

## tracking 브랜치

`-u` (upstream)으로 로컬·원격 브랜치를 연결한다.

```bash
git push -u origin feature/search-bar
# 이후부터 git push / git pull 만으로 동기화
```

```bash
git branch -vv   # tracking 관계 확인
```

## 브랜치 보호 규칙 (GitHub)

팀 저장소에서 main에 흔히 적용하는 규칙:

- **직접 push 금지** — PR만 허용
- **리뷰 1명 이상** 승인 필요
- **CI 통과** 필수 (lint, test, build)
- **force push 금지**

프론트엔드 CI 예시:

```text
PR → ESLint → TypeScript check → Vitest → Build → merge
```

## 언제 브랜치를 나눌까?

| 나누기 좋음 | 한 브랜치에 묶어도 됨 |
| --- | --- |
| 독립 기능 (로그인, 결제) | 오타 수정 |
| 리뷰 단위가 큰 작업 | 같은 기능의 작은 UI 조정 |
| 실험적 작업 (나중에 버릴 수도) | 연속된 커밋으로 나누기 어려운 긴급 hotfix |

## 정리

| 개념 | 내용 |
| --- | --- |
| feature 브랜치 | 기능 개발용, main에서 분기 |
| 네이밍 | `feature/`, `fix/`, `hotfix/` 접두사 |
| Trunk-based | 짧은 브랜치, main 자주 merge |
| 보호 규칙 | main 직접 push 금지 + CI |
