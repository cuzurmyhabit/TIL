# 머지와 리베이스

두 브랜치의 변경을 합치는 대표적인 방법이다. 프론트엔드 팀에서는 **팀 규칙**에 따라 하나를 표준으로 정한다.

---

## Merge (머지)

한 브랜치의 변경을 다른 브랜치에 **합친다**. merge 커밋이 생길 수 있다.

```bash
git checkout main
git merge feature/login-ui
```

### Fast-forward merge

main이 feature 시작 이후 변경이 없으면 **직선으로 이동**한다.

```text
Before:
main:    A --- B
              \
feature:       C --- D

After (fast-forward):
main:    A --- B --- C --- D
```

```bash
git merge feature/login-ui   # 자동 fast-forward
```

### 3-way merge (merge commit)

양쪽 모두 새 커밋이 있으면 **merge commit**이 생긴다.

```text
Before:
main:    A --- B --- E
              \
feature:       C --- D

After:
main:    A --- B --- E --- M (merge commit)
              \           /
feature:       C --- D ---
```

```bash
git merge feature/login-ui
# 충돌 없으면 에디터에서 merge 메시지 확인 후 저장
```

### --no-ff 옵션

fast-forward를 막고 **항상 merge commit**을 만든다. 히스토리에서 기능 단위가 보인다.

```bash
git merge --no-ff feature/login-ui
```

---

## Rebase (리베이스)

내 커밋을 **다른 브랜치 끝에 다시 쌓는다**. 히스토리가 직선으로 깔끔해진다.

```bash
git checkout feature/login-ui
git rebase main
```

```text
Before:
main:    A --- B --- C
              \
feature:       D --- E

After rebase:
main:    A --- B --- C
                      \
feature:               D' --- E'
```

`D'`, `E'`는 내용은 같지만 **새 해시**의 커밋이다.

### main 반영할 때 (실무 패턴)

```bash
git checkout feature/my-work
git fetch origin
git rebase origin/main
# 충돌 해결 후
git push --force-with-lease
```

feature 브랜치 작업 중 main이 앞서 나갔을 때 자주 쓴다.

---

## Merge vs Rebase 비교

| | Merge | Rebase |
| --- | --- | --- |
| **히스토리** | merge commit 포함, 그래프 복잡 | 직선, 깔끔 |
| **안전성** | 기존 커밋 유지 | 커밋 해시 변경 |
| **충돌** | merge 시 1회 | 커밋마다 발생 가능 |
| **팀 규칙** | main merge에 흔함 | feature 정리·PR 전에 흔함 |

### ⭐ 황금 규칙

> **이미 push해서 다른 사람이 쓰는 브랜치에는 rebase하지 않는다.**

공유 브랜치(main, develop)에 rebase하면 팀원 히스토리가 꼬인다.

| 상황 | 권장 |
| --- | --- |
| main에 feature merge | merge (GitHub PR merge) |
| 내 feature에 main 최신 반영 | rebase 또는 merge (팀 규칙) |
| PR 전 커밋 정리 | interactive rebase (로컬) |

## GitHub PR merge 옵션

PR merge 시 GitHub가 제공하는 3가지:

| 옵션 | 결과 |
| --- | --- |
| **Merge commit** | merge commit 생성 |
| **Squash and merge** | PR 커밋을 **하나로 압축** 후 merge |
| **Rebase and merge** | PR 커밋을 main 위에 rebase |

### Squash merge — 프론트엔드 팀에서 인기

```text
feature:  "WIP" → "fix typo" → "feat: 모달 추가" → "리뷰 반영"
                    ↓ squash
main:     "feat: 사용자 프로필 모달 추가 (#42)"
```

- main 히스토리가 깔끔
- WIP·typo 커밋이 main에 안 남음
- 단, feature 브랜치의 세부 커밋 이력은 main에서 사라짐

---

## Interactive Rebase (로컬 정리)

PR 올리기 전 커밋을 정리할 때:

```bash
git rebase -i HEAD~3   # 최근 3개 커밋
```

에디터에서:

```text
pick a1b2c3 feat: 모달 UI
squash d4e5f6 fix: 오타
squash g7h8i9 리뷰 반영

# pick → 유지
# squash → 이전 커밋에 합침
# reword → 메시지 수정
# drop → 커밋 삭제
```

**push 전, 혼자 쓰는 브랜치**에서만 사용한다.

---

## 정리

| 명령어 | 용도 |
| --- | --- |
| `git merge` | 브랜치 합치기 |
| `git rebase` | 커밋을 base 위에 재배치 |
| Squash merge | PR 커밋을 하나로 압축 |
| `git rebase -i` | 로컬 커밋 정리 |
