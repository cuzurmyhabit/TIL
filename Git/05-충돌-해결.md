# 충돌 해결

두 사람이 **같은 파일의 같은 부분**을 수정하면 Git이 자동으로 합치지 못하고 **충돌(conflict)** 이 발생한다.  
프론트엔드에서는 `package.json`, 공통 컴포넌트, 스타일 파일에서 자주 일어난다.

---

## 충돌이 나는 상황

```text
main:    Header.tsx — 로고 변경 (A팀)
feature: Header.tsx — 네비 메뉴 추가 (B팀)
         → merge 시 충돌!
```

```bash
git merge main
# Auto-merging src/components/Header.tsx
# CONFLICT (content): Merge conflict in src/components/Header.tsx
```

---

## 충돌 마커 읽기

Git이 충돌 파일에 마커를 삽입한다.

```tsx
<<<<<<< HEAD
<nav className="new-nav">메뉴 v2</nav>
=======
<nav className="old-nav">메뉴 v1</nav>
>>>>>>> feature/header-update
```

| 마커 | 의미 |
| --- | --- |
| `<<<<<<< HEAD` | 현재 브랜치(merge 받는 쪽) 내용 시작 |
| `=======` | 구분선 |
| `>>>>>>> branch-name` | 합치려는 브랜치 내용 끝 |

**마커를 포함한 최종 코드**를 직접 작성해야 한다.

```tsx
// ✅ 해결 후
<nav className="new-nav">메뉴 v2</nav>
```

## 충돌 해결 절차

```bash
# 1. 충돌 파일 확인
git status

# 2. 에디터에서 마커 제거 + 올바른 코드 작성
# VS Code: "Accept Current" / "Accept Incoming" / "Accept Both" 버튼 제공

# 3. 해결된 파일 스테이징
git add src/components/Header.tsx

# 4. merge 완료
git commit   # merge 중이면 메시지 자동 생성
# rebase 중이면:
git rebase --continue
```

### merge 중단

```bash
git merge --abort    # merge 전 상태로 복귀
git rebase --abort   # rebase 전 상태로 복귀
```

## 프론트엔드에서 자주 충돌나는 파일

| 파일 | 원인 | 팁 |
| --- | --- | --- |
| `package-lock.json` | 의존성 각자 설치 | 한쪽 기준으로 재생성 |
| `package.json` | 패키지 추가 시점 다름 | 수동 병합 후 `npm install` |
| 공통 컴포넌트 | 동시 수정 | 팀과 영역 분리 |
| `*.css` / Tailwind | 같은 클래스 수정 | 디자인 시스템으로 분리 |
| 라우트 설정 | 페이지 동시 추가 | 파일/폴더 단위로 분리 |

### package-lock.json 충돌

```bash
# 1. package.json 수동 병합
# 2. lock 파일 삭제 후 재생성
rm package-lock.json
npm install
git add package.json package-lock.json
```

팀 규칙으로 **lock 파일 충돌 시 한쪽 버전 채택 + npm install**이 정해져 있는 경우도 많다.

## VS Code에서 충돌 해결

Source Control 패널에서 충돌 파일에 **Merge Editor**가 열린다.

- **Accept Current Change** — 내 브랜치(HEAD) 유지
- **Accept Incoming Change** — 합치려는 브랜치 내용 채택
- **Accept Both Changes** — 둘 다 유지 (순서 조정 필요할 수 있음)
- **Compare Changes** — diff로 비교

## 충돌 예방법

### 1. 작은 PR, 자주 merge

```text
❌ 2주짜리 거대 feature 브랜치
✅ 1~2일 단위로 main에 반영
```

### 2. 작업 시작 전 main 반영

```bash
git fetch origin
git rebase origin/main   # 또는 merge
```

### 3. 파일·폴더 단위로 작업 분리

```text
A: src/features/auth/
B: src/features/cart/
→ 같은 파일을 안 건드리면 충돌 없음
```

### 4. 공통 컴포넌트 변경은 사전 공유

## rebase 중 충돌

```bash
git rebase main
# CONFLICT ...

# 파일 수정 후
git add <파일>
git rebase --continue

# 여러 커밋에 걸쳐 충돌이 반복될 수 있음
# 포기하려면
git rebase --abort
```

merge보다 rebase 충돌이 **커밋마다** 발생할 수 있어 번거로울 때가 있다.

---

## 정리

| 단계 | 명령어/행동 |
| --- | --- |
| 충돌 확인 | `git status` |
| 코드 수정 | 마커 제거, 올바른 코드 작성 |
| 완료 | `git add` → `git commit` 또는 `rebase --continue` |
| 포기 | `merge --abort` / `rebase --abort` |

충돌은 실패가 아니라 **협업의 자연스러운 과정**이다. 당황하지 말고 diff를 차분히 읽으면 된다.
