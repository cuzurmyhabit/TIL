# 커밋과 일상 작업

프론트엔드 개발에서 가장 자주 반복하는 Git 작업 흐름을 정리한다.

---

## 기본 작업 사이클

```bash
# 1. 현재 상태 확인
git status

# 2. 변경 파일 스테이징
git add src/components/Button.tsx

# 3. 커밋
git commit -m "feat: 버튼 hover 스타일 추가"

# 4. 원격에 push
git push
```

```text
수정 → add → commit → push
```

---

## git add

### 특정 파일만

```bash
git add src/App.tsx
git add src/components/Header.tsx src/components/Footer.tsx
```

### 폴더 전체

```bash
git add src/
```

### 모든 변경 (주의해서 사용)

```bash
git add .
```

### 대화형으로 선택

```bash
git add -p    # hunk 단위로 스테이징 여부 선택
```

하나의 파일에 **서로 다른 기능 수정**이 섞였을 때 유용하다.

### 스테이징 취소

```bash
git restore --staged <파일>   # add 취소 (파일 내용은 유지)
```

## git commit

### 기본 커밋

```bash
git commit -m "fix: 모바일에서 메뉴 안 열리는 버그 수정"
```

### ⭐ 좋은 커밋 메시지 작성법

```text
<type>: <한 줄 요약>

(선택) 본문 — 왜 이 변경이 필요했는지
```

| type | 용도 | 예시 |
| --- | --- | --- |
| `feat` | 새 기능 | `feat: 다크모드 토글 추가` |
| `fix` | 버그 수정 | `fix: 이미지 lazy load 오류` |
| `style` | 포맷·스타일 (로직 변경 X) | `style: Prettier 적용` |
| `refactor` | 리팩터링 | `refactor: useAuth 훅 분리` |
| `chore` | 빌드·설정 | `chore: eslint 규칙 업데이트` |
| `docs` | 문서 | `docs: README 설치 가이드 추가` |
| `test` | 테스트 | `test: LoginForm 단위 테스트` |

### 커밋 단위 원칙

- **하나의 커밋 = 하나의 논리적 변경**
- `package.json` 수정 + UI 수정 + 리팩터링을 한 커밋에 넣지 않기
- 리뷰어가 diff를 읽기 쉬운 크기로

```bash
# ❌
git commit -m "여러 작업"

# ✅ 기능별로 나누기
git commit -m "feat: 로그인 폼 UI 추가"
git commit -m "feat: 로그인 API 연동"
```

### 직전 커밋 메시지 수정 (push 전에만!)

```bash
git commit --amend -m "fix: 오타 수정된 메시지"
```

이미 push한 커밋을 amend하면 히스토리가 바뀌어 **팀원과 충돌**할 수 있다.

## git push / git pull

### push

```bash
git push                    # 현재 브랜치를 tracking 브랜치에 push
git push -u origin feature/login   # 최초 push 시 upstream 설정
```

### pull

```bash
git pull    # 원격 변경을 가져와 merge (또는 rebase)
```

작업 시작 전 습관:

```bash
git checkout main
git pull
git checkout feature/my-work
git merge main    # 또는 git rebase main
```

main 최신 상태를 자주 반영하면 충돌 규모가 작아진다.

## 변경 내용 확인

```bash
git diff                  # unstaged 변경
git diff --staged         # staged 변경 (다음 커밋 내용)
git diff main..feature/x  # 브랜치 간 차이
git show <커밋해시>        # 특정 커밋 상세
```

PR 올리기 전에 `git diff main`으로 전체 변경을 한번 훑는 습관이 좋다.

## 작업 임시 저장 — git stash

다른 브랜치로 급히 이동해야 할 때:

```bash
git stash              # 현재 변경 임시 저장
git checkout hotfix/urgent-bug
# ... 작업 ...
git checkout feature/my-work
git stash pop          # stash 복원 + 목록에서 제거
```

```bash
git stash list         # stash 목록
git stash apply        # 복원하되 stash 유지
git stash drop         # stash 삭제
```

`node_modules` 변경 등은 stash되지 않는다. 보통 소스 코드 변경에 쓴다.

## 파일 되돌리기 (작업 중)

```bash
# 특정 파일을 마지막 커밋 상태로 (주의: 변경 사라짐)
git restore src/App.tsx

# 모든 unstaged 변경 되돌리기
git restore .
```

## 프론트엔드 일상 시나리오

### 시나리오 1: 컴포넌트 작업 후 push

```bash
git checkout -b feature/user-card
# ... UserCard.tsx 작성 ...
git add src/components/UserCard.tsx
git commit -m "feat: UserCard 컴포넌트 추가"
git push -u origin feature/user-card
```

### 시나리오 2: 실수로 main에서 작업함

```bash
git stash
git checkout -b feature/actual-work
git stash pop
```

### 시나리오 3: package-lock.json만 바뀜

```bash
git status   # lock 파일만 변경 확인
git add package-lock.json
git commit -m "chore: lock file 업데이트"
```

## 정리

| 명령어 | 용도 |
| --- | --- |
| `git add` | 스테이징 |
| `git commit -m` | 커밋 |
| `git push` / `git pull` | 원격 동기화 |
| `git diff` | 변경 확인 |
| `git stash` | 임시 저장 |
| `git restore` | 파일 되돌리기 |
